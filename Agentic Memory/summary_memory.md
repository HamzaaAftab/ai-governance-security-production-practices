## Summary Memory
- Instead of sending whole conversation history, we can send the summary of the conversation history to the LLM.

- Instead of discarding the history, it summarizes it or compresses it. It preserves the context and make a summary of it.

#### The Main Idea:
- When the conversation grows too long, use the LLM itself to compress the oldest turns into a concise summary - then replace those raw turns with the summary in the context window.  
- In most of the production systems, summary memory is used alot. 
- Claude code uses a feature calls as compaction which does this as well.
- You are saving on token and at the same time preserving the context.
 
 ### The summary is generated by the LLM, not hand-coded
 - The model produces a compact paragraph. That paragraph replaces the raw turns. This is called Abstractive Summarisation. 

 ## Summarisation is triggered by threshold, not on every turn
 - You do not summarise after every message - that would be expensive and slow. You define a trigger
 - Turn Based
 - Token Based

 ## Lossy compression is the fundamental risk
 - Losing out of complete information.
 - The model decides what matters and it can or might get wrong. 
 - In third compression cycle, the information of first cycle might get lose and so on. 
 - Your summarization prompt should be well designed. 
 - You can chain the summaries instead of making a summary of all the history. 


## Production Verdict:
- Strong production pattern - widely used, but must be engineered carefully. 
- It is used in claude code, kiro cli and most long running agent frameworks. 
- The risk is information loss on the third or fourth compression cycle.
- The mitigation s a domain specific summarisation prompt that explicitly names the categories of facts that must survive.
- Never use a generic sumarry prompt in a domain with critical facts.
